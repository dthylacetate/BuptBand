# 架构

**主要目标：** 确定技术栈，确保系统的高移植性与开发效率。

------

## 1. 技术栈概览 

| **层次**     | **选型**            | **说明**                                                     |
| ------------ | ------------------- | ------------------------------------------------------------ |
| **后端框架** | **Spring Boot 3.x** | Java 界的工业标准，起步快，配置简单，适合 MVP 开发。         |
| **构建工具** | **Apache Maven**    | 依赖管理成熟，适合在复杂的网络环境（如校园网）下配置镜像。   |
| **数据库**   | **H2 Database**     | **文件模式存储**。极其轻量，支持存放在项目目录下，移植性极佳。 |
| **ORM 框架** | **Spring Data JPA** | 屏蔽底层 SQL 差异。未来如果要从 H2 升级到 MySQL，只需改配置。 |
| **前端框架** | **Vue 3 + Vite**    | 响应式开发首选，配合微型服务器即可在内测阶段提供流畅体验。   |
| **UI 方案**  | **Tailwind CSS**    | 快速实现“乐手名片”等高定制化的 UI，无需编写大量冗余 CSS。    |

------

## 2. 存储方案与移植策略

为了方便移植，采用 **H2 Embedded File Mode**：

- **存储位置：** 数据库文件将配置在项目根目录下的 `/data` 文件夹中。

- **移植方法：** 1.  打包生成 `jar` 文件。

  \2.  连同 `/data` 文件夹（包含 `.db`）直接拷贝至新机器。

  \3.  在新机器运行 `java -jar app.jar` 即可。

------

## 3. 系统逻辑架构 

项目采用经典的 **三层架构 ，确保代码整洁且易于维护：

1. **Web/Controller Layer:** 负责接收前端请求，处理参数校验。
2. **Service Layer:** 负责业务逻辑（例如：判断一个乐手是否已经发布过招募）。
3. **Repository/DAO Layer:** 负责通过 JPA 与 H2 文件进行数据交互。

------

## 4. 关键特性设计

- **数据库版本控制 :** 即使是 H2，我们也引入 Flyway。建表语句（DDL）会以 SQL 脚本形式存在代码里。当迁移到空环境时，程序启动会自动在本地生成数据库结构，无需手动导数据。

- **前后端分离:**

  后端只输出 JSON 接口。

------

## 5. 环境要求 (Prerequisites)

- **JDK:** 17 或 21 (LTS 版本)
- **IDE:** IntelliJ IDEA
- **Database Tool:** IDEA 内置的 Database 浏览器（支持 H2 可视化）
